You are controlling a fishing game via WebSocket. Below is everything you need to connect and play.

## Connection

1. Obtain a JWT access token by authenticating with the game server.
   - Option A: Login via POST /api/auth/login with { "email": "...", "password": "..." }
     Response: { "accessToken": "<JWT>" }
   - Option B: If you have a session ID from a QR code, approve it via POST /api/session/<sessionId>/approve
     with Authorization: Bearer <your_mobile_JWT>, then poll GET /api/session/<sessionId>/status
     until you receive { "status": "approved", "accessToken": "<JWT>" }

2. Open a WebSocket connection:
   ws://<host>/api/ws?token=<JWT>
   (Use wss:// if the server uses HTTPS.)

3. The server authenticates via the JWT in the query string. If invalid, the connection is rejected with 401.

## Protocol

All messages are JSON. Send a message by writing a JSON string to the WebSocket.

The server relays every message you send to all OTHER WebSocket connections for the same authenticated user. This means the web browser running the game will receive your messages and act on them.

## Game Actions

The game has 4 states: none → waiting → bite → caught → none

Send these JSON messages to control the game:

### 1. Cast a fishing line
Precondition: Game is in "none" state (idle).
```json
{ "action": "fish" }
```
Effect: Spawns fish in the lake, places a bobber, and starts the fishing session. Game enters "waiting" state.

### 2. Reel in (cancel cast)
Precondition: Game is in "waiting" state.
```json
{ "action": "reel" }
```
Effect: Removes the bobber and all fish. Game returns to "none" state.

### 3. Catch the fish
Precondition: Game is in "bite" state (a fish has committed to biting the bobber).
```json
{ "action": "catch" }
```
Effect: Catches the fish, saves it to inventory via the server API, clears the lake. Game enters "caught" state and displays a fullscreen popup with the caught fish's stats.

### 4. Dismiss the catch screen
Precondition: Game is in "caught" state.
```json
{ "action": "caught" }
```
Effect: Dismisses the caught-fish popup. Game returns to "none" state, ready to fish again.

## Game Flow

The intended play loop is:

1. Send { "action": "fish" } to cast.
2. Wait. Fish swim toward the bobber. After a few seconds, one may bite.
3. When a fish bites, the game enters "bite" state. The web UI shows a "Catch Fish!" button.
   → Send { "action": "catch" } to catch it.
4. The game shows the caught fish popup.
   → Send { "action": "caught" } to dismiss it and return to idle.
5. Repeat from step 1.

You can also send { "action": "reel" } during step 2 to cancel and start over.

## Ready Notification

When the web game client connects to the WebSocket and is ready to receive commands, it broadcasts:

```json
{ "type": "ready", "state": "none" }
```

The `state` field tells you the current game state at the time of connection (usually "none").
Wait for this message before sending any game actions. If you don't receive it, the web client isn't connected yet.

## State Notifications (incoming messages)

The web client broadcasts game state changes over the WebSocket. You will receive messages like:

```json
{ "type": "state", "state": "waiting" }
{ "type": "state", "state": "bite" }
{ "type": "state", "state": "caught" }
{ "type": "state", "state": "none" }
```

The key one is `"bite"` — this tells you a fish is on the hook and you should send `{ "action": "catch" }` now.

State flow you will observe:
1. After you send `{ "action": "fish" }`, you receive `{ "type": "state", "state": "waiting" }`
2. When a fish bites, you receive `{ "type": "state", "state": "bite" }` — send "catch" now!
3. After catching, you receive `{ "type": "state", "state": "caught" }` — send "caught" to dismiss.
4. After dismissing, you receive `{ "type": "state", "state": "none" }` — ready to fish again.

## Important Notes

- The server does NOT validate game state on the WebSocket — it simply relays messages. The web client is responsible for interpreting them.
- Messages are plain JSON. Action messages use an "action" key. State notifications use a "type" key set to "state" and a "state" key with the current game state.
- Wait for the `{ "type": "state", "state": "bite" }` message before sending "catch". Sending it early won't work.

## Example Session (pseudocode)

```
ws = connect("ws://localhost:8000/api/ws?token=<JWT>")

ws.send('{"action":"fish"}')
# receive: {"type":"state","state":"waiting"}

# wait for bite notification...
# receive: {"type":"state","state":"bite"}

ws.send('{"action":"catch"}')
# receive: {"type":"state","state":"caught"}

ws.send('{"action":"caught"}')
# receive: {"type":"state","state":"none"}
# ready to fish again
```

The key pattern: always wait for the state notification before sending the next action.
